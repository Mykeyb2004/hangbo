# 主流程目录约定与默认配置化设计

## 1. 背景

当前仓库已经具备完整的数据分析业务链路：

- 原始数据预查错
- 原始数据预处理
- 满意度分项统计
- 满意度汇总
- 样本汇总
- PPT 生成

但当前运行方式仍然以“批次级配置文件 + 多个脚本入口”为主，存在以下问题：

1. `job*.toml`、`report_jobs*.toml`、GUI 运行时 TOML 中反复描述输入输出路径。
2. 路径型参数占比过高，日常跑批时重复填写成本高。
3. 业务人员真正需要关注的是“哪一年的哪个批次”，而不是每一步脚本的目录拼装细节。
4. 当前代码已有主链，但缺少一个稳定的 CLI 主流程入口来串起“查错 -> 人工修正 -> 继续执行”。

本设计的目标，是在**不重写现有业务计算逻辑**的前提下，将仓库升级为：

- 使用固定目录约定
- 使用一份全局默认配置
- 使用一个主流程 CLI 命令
- 支持查错后暂停，人工修改后在当前运行中确认继续

## 2. 本次设计范围

### 2.1 目标

Phase 1 的目标是提供一个新的主流程命令：

```bash
uv run python main_pipeline.py --year 2026 --batch 3月
```

该命令完成以下事情：

1. 根据固定目录约定自动定位输入输出路径
2. 执行预查错
3. 如果发现阻断问题，则暂停等待人工修改
4. 用户确认后重新预查错
5. 若通过，则自动继续执行：
   - `survey_stats.py`
   - `summary_table.py`
   - `sample_table.py`
   - `generate_ppt.py`

### 2.2 非目标

本阶段不做以下事情：

1. 不重写 `survey_stats.py`、`summary_table.py`、`sample_table.py`、`generate_ppt.py` 的核心业务算法
2. 不移除旧的脚本入口和旧的批次级 TOML 跑法
3. 不优先改造 GUI
4. 不实现复杂断点续跑
5. 不自动替用户修改原始 Excel 数据

## 3. 已确认设计决策

以下事项已在讨论中确认：

1. 使用一个总根目录：`data`
2. 满意度产物拆分为两类目录：
   - `satisfaction_detail`
   - `satisfaction_summary`
3. 样本汇总单独放目录：
   - `sample_summary`
4. PPT 单独放目录：
   - `ppt`
5. 原始数据目录固定为：
   - `raw`
6. 批次维度保留“月度批次”和“组合批次”两类正式目录
7. 原始文件名采用强制标准名
8. 预查错后停下，等待用户确认再继续
9. 第一期只做 CLI，不改 GUI 主流程
10. 保留一份全局默认配置文件，移除按批次写路径的配置文件

## 4. 方案比较

### 4.1 方案 A：新增主流程编排层，保留现有引擎

做法：

- 新增 `main_pipeline.py`
- 新增统一目录解析和默认配置层
- 通过 import 的方式复用现有脚本内部函数

优点：

- 改动边界清晰
- 对现有业务算法风险最低
- 最容易逐步替代现有批次配置文件

缺点：

- 短期内会出现“新主流程入口 + 旧脚本入口并存”

### 4.2 方案 B：让每个脚本都全面目录化

做法：

- 直接改造所有脚本，使其默认按 `data/...` 目录工作

优点：

- 从脚本层面看更统一

缺点：

- 改动面大
- 回归风险高
- 更难快速形成稳定的一键主流程

### 4.3 方案 C：重写为单体流水线

做法：

- 弱化现有脚本边界，重写为单个大型流水线程序

优点：

- 理论上最终形态最整齐

缺点：

- 成本高
- 不适合当前仓库阶段
- 风险远超收益

### 4.4 结论

采用 **方案 A**。

也就是：

- 新增稳定的 CLI 主流程入口
- 保留现有业务引擎
- 把路径型配置收敛到目录约定
- 把必要业务参数收敛到一份全局默认配置

## 5. 目标目录模型

### 5.1 总目录结构

根目录固定为：

```text
data/
  raw/
  satisfaction_detail/
  satisfaction_summary/
  sample_summary/
  ppt/
```

### 5.2 年份与批次路径

所有业务目录统一采用：

```text
data/<kind>/<year>/<batch>/
```

示例：

```text
data/raw/2026/3月/
data/raw/2026/1-2月/
data/raw/2026/Q1/

data/satisfaction_detail/2026/3月/
data/satisfaction_summary/2026/3月/
data/sample_summary/2026/3月/
data/ppt/2026/3月/
```

### 5.3 批次命名规则

本阶段不对批次名做过度限制，只要求：

1. 作为路径最后一级目录存在
2. 能够被用户显式传入 `--batch`
3. 支持月度批次和组合批次，例如：
   - `3月`
   - `1-2月`
   - `Q1`

程序不自动推导组合批次含义，不负责把 `Q1` 自动拆解为月度集合。  
组合批次在目录层面被视为正式批次目录。

## 6. 原始数据文件命名约定

### 6.1 标准文件名

`data/raw/{year}/{batch}` 下原始文件采用固定标准名：

- `展览.xlsx`
- `会议.xlsx`
- `酒店.xlsx`
- `餐饮.xlsx`
- `会展服务商.xlsx`
- `旅游.xlsx`

### 6.2 设计选择

本阶段不支持：

- 任意文件名自动猜测
- 每个批次再配一个轻量映射文件

原因：

1. 当前业务的稳定性优先于灵活性
2. 现有 `survey_customer_category_rules.py` 已按标准来源文件名组织
3. 使用固定标准文件名，可最大化减少路径和映射配置

### 6.3 校验方式

主流程预查错阶段会检查：

- 原始批次目录是否存在
- 目录中是否存在标准文件名

缺失标准来源文件属于**阻断问题**。

## 7. 输出目录与命名约定

### 7.1 满意度分项结果

目录：

```text
data/satisfaction_detail/{year}/{batch}/
```

内容：

- 每个客户类型一个 `xlsx`
- 文件名继续沿用现有输出逻辑，例如：
  - `展览主承办.xlsx`
  - `参展商.xlsx`
  - `酒店餐饮客户.xlsx`

### 7.2 满意度汇总表

目录：

```text
data/satisfaction_summary/{year}/{batch}/
```

文件名固定为：

```text
{batch}客户类型满意度汇总表.xlsx
```

### 7.3 样本汇总表

目录：

```text
data/sample_summary/{year}/{batch}/
```

文件名固定为：

```text
{batch}客户类型样本统计表.xlsx
```

### 7.4 PPT 结果

目录：

```text
data/ppt/{year}/{batch}/
```

文件名固定为：

```text
{batch}满意度报告.pptx
```

### 7.5 目录创建策略

若输出目录不存在，主流程自动创建。  
这类情况属于**警告级自动修复**，不会阻断流程。

## 8. 默认配置模型

### 8.1 配置目标

全局默认配置只保留真正业务不能缺的参数，不再保留每个批次的路径型配置。

### 8.2 配置文件位置

新增一份全局默认配置文件：

```text
pipeline.defaults.toml
```

建议放在项目根目录。

### 8.3 计划保留的默认配置项

```toml
sheet_name = "问卷数据"
calculation_mode = "template"

[ppt]
template_path = "templates/template.pptx"
sheet_name_mode = "first"
section_mode = "auto"
blank_display = ""
title_suffix = ""
max_single_table_rows = 18
max_split_table_rows = 19
sort_files = true
body_font_size_pt = 10.5
header_font_size_pt = 11.0
summary_font_size_pt = 12.0
template_slide_index = 0

[ppt.chart_page]
enabled = true
placeholder_text = "图表分析内容待补充。后续将在此处补充该客户分组二级指标的整体解读、优势项与待提升项。"
image_dpi = 220

[ppt.llm_notes]
enabled = true
env_path = ".env"
system_role_path = "system_role.md"
target_chars = 300
temperature = 0.6
max_tokens = 500
checkpoint_chars = 80
```

### 8.4 不再放入默认配置的内容

以下内容不再需要在配置里写：

- 当前批次原始目录路径
- 当前批次分项输出目录
- 当前批次汇总输出目录
- 当前批次样本输出目录
- 当前批次 PPT 输出目录
- 当前批次专属 `job*.toml`
- 当前批次专属 `report_jobs*.toml`

这些内容全部由：

- `--year`
- `--batch`
- 固定目录约定

共同推导。

### 8.5 与年份相关的特殊约束

`sample_table.py` 中涉及的默认年份，不再在全局配置文件中固定写死，而是由主流程直接使用 CLI 的 `--year` 参数传入。

这样做的原因是：

1. 避免配置文件中的年份与当前批次年份冲突
2. 保证 `main_pipeline.py --year 2026 --batch 3月` 这一类命令天然自洽
3. 降低“默认配置改一年、忘记切回去”的人为风险

## 9. CLI 入口设计

### 9.1 命令形式

```bash
uv run python main_pipeline.py --year 2026 --batch 3月
```

### 9.2 参数设计

主流程 CLI 只保留最小必要参数：

| 参数 | 是否必填 | 含义 |
| --- | --- | --- |
| `--year` | 是 | 年份目录，例如 `2026` |
| `--batch` | 是 | 批次目录，例如 `3月`、`1-2月`、`Q1` |
| `--config` | 否 | 全局默认配置文件路径，默认 `pipeline.defaults.toml` |

Phase 1 不再要求命令行传各类输入输出路径。

## 10. 主流程交互设计

### 10.1 运行流程

主流程固定按以下顺序运行：

1. 解析批次路径
2. 执行预查错
3. 若存在阻断问题，则打印问题清单并暂停
4. 等待用户确认
5. 用户确认后，重新执行一次预查错
6. 若预查错通过，则继续执行：
   - 单月批次按需自动补写 `年份` / `月份`
   - 满意度分项统计
   - 满意度汇总
   - 样本汇总
   - PPT 生成

### 10.2 为什么“确认后要重新预查错”

确认后不直接继续，而是再次执行预查错，原因是：

1. 避免用户修改不完整仍继续出结果
2. 保证“继续执行”的前提是当前数据已经通过口径检查
3. 降低主流程在错误数据上继续运行的风险

### 10.3 终端确认方式

主流程暂停后，在终端提示用户：

- 已发现阻断问题
- 请修改 `data/raw/{year}/{batch}` 下原始数据
- 修改完成后输入确认指令继续

确认输入接受以下值之一：

- `y`
- `yes`
- `继续`

其他输入一律视为“未确认”，继续等待或允许用户中止。

### 10.4 典型交互

```text
[PRECHECK] 发现阻断问题：
- 餐饮.xlsx 中存在未映射标签组合
- 会议.xlsx 缺少标准来源数据列

请先修改原始数据目录：
data/raw/2026/3月

修改完成后输入 y / yes / 继续 以重新检查并继续执行：
> 继续

[PRECHECK] 正在重新检查...
[PRECHECK] 已通过
[PIPELINE] 检测为单月批次，正在补写年份/月份...
[PIPELINE] 开始生成满意度分项统计...
...
[PIPELINE] PPT 已生成
```

## 11. 阻断与警告规则

### 11.1 阻断问题

以下问题属于阻断问题，必须暂停：

1. `data/raw/{year}/{batch}` 不存在
2. 标准来源文件缺失
3. `check_unmapped_customer_records.py` 检测到未映射标签或未映射标签组合
4. 关键 sheet 缺失
5. 预查错过程自身报错
6. 对于组合批次，如果原始数据缺少可用于样本分组的 `年份` / `月份` 字段，则阻断样本汇总流程

### 11.2 警告问题

以下问题属于警告，可继续：

1. 自动执行了 `phase_column_preprocess.py` 兼容处理
2. 输出目录不存在但已自动创建
3. 对于单月批次，年月字段缺失但可由主流程自动补写

### 11.3 规则原则

原则如下：

- **可能影响统计口径正确性的问题，一律阻断**
- **程序能够自动修复且不会改变业务口径的问题，作为警告**

### 11.4 年月补写策略

主流程对 `fill_year_month_columns.py` 的使用规则明确如下：

1. **单月批次**
   - 当批次名能唯一解析为一个月份时，例如 `3月`
   - 主流程允许自动补写：
     - `年份 = --year`
     - `月份 = 3`
   - 这属于警告级自动修复

2. **组合批次**
   - 例如 `1-2月`、`Q1`
   - 主流程不自动批量写入统一月份值
   - 因为组合批次目录中的不同文件或不同行记录可能对应不同月份
   - 若这类批次缺少 `年份` / `月份` 信息，主流程将其视为阻断问题，由用户先修正原始数据

3. **解析失败的批次名**
   - 若批次名既不是明确单月，也不是现有组合批次
   - 主流程不尝试自动补写月份
   - 只保留原数据

## 12. 现有脚本复用策略

### 12.1 保留为业务引擎的脚本

以下脚本继续保留，并作为主流程的底层能力：

- `survey_stats.py`
- `summary_table.py`
- `sample_table.py`
- `generate_ppt.py`
- `check_unmapped_customer_records.py`

### 12.2 主流程调用方式

主流程优先采用 **import 函数调用**，而不是 `subprocess` 串脚本。

原因：

1. 更容易复用默认配置对象
2. 更容易做细粒度错误分类
3. 更容易实现“暂停 -> 确认 -> 继续”的交互
4. 更容易编写单元测试

### 12.3 允许保留的旧入口

现有脚本的 CLI 入口暂时不移除，作为兼容入口保留。

## 13. 新增模块设计

### 13.1 `main_pipeline.py`

职责：

- 解析 CLI 参数
- 加载全局默认配置
- 构建批次上下文
- 控制预查错、暂停确认、重新检查和后续执行
- 输出统一日志

### 13.2 `pipeline_paths.py`

职责：

- 根据 `year + batch` 统一推导输入输出目录
- 统一推导标准文件路径
- 统一推导汇总文件、样本文件和 PPT 文件路径

### 13.3 `pipeline_config.py`

职责：

- 读取 `pipeline.defaults.toml`
- 解析并校验默认配置
- 输出供主流程和 PPT/样本环节使用的结构化配置对象

### 13.4 `pipeline_precheck.py`

职责：

- 封装目录存在性检查
- 封装标准来源文件检查
- 复用 `check_unmapped_customer_records.py` 的目录核查能力
- 输出阻断问题与警告问题列表

### 13.5 `pipeline_models.py`

职责：

- 定义批次上下文
- 定义阻断问题、警告问题、步骤结果等数据结构

## 14. 主流程步骤细化

### 14.1 路径解析

输入：

- `year`
- `batch`

输出：

- 原始数据目录
- 分项结果目录
- 汇总结果目录
- 样本结果目录
- PPT 结果目录
- 日志目录

### 14.2 预查错

执行内容：

1. 检查原始目录存在性
2. 检查标准来源文件
3. 执行未映射记录核查
4. 汇总阻断与警告

### 14.3 等待确认

若存在阻断问题：

1. 输出问题清单
2. 提示用户修改原始目录
3. 阻塞等待确认

### 14.4 重新预查错

收到确认后：

1. 重新执行与首次相同的预查错逻辑
2. 若仍有阻断问题，则继续停下
3. 若通过，则进入正式执行

### 14.5 单月批次年月补写

若批次名可明确解析为单月，例如 `3月`，则在正式执行前：

1. 使用 `--year` 作为年份值
2. 使用解析出的月份值作为月份文本
3. 对原始批次目录下标准来源文件执行补写

若为组合批次，则跳过本步骤。

### 14.6 满意度分项统计

使用：

- 目录模式
- 固定原始数据目录
- 固定输出目录
- 全局默认 `calculation_mode`

### 14.7 满意度汇总

输入：

- `data/satisfaction_detail/{year}/{batch}`

输出：

- `data/satisfaction_summary/{year}/{batch}/{batch}客户类型满意度汇总表.xlsx`

### 14.8 样本汇总

输入：

- `data/raw/{year}/{batch}`

输出：

- `data/sample_summary/{year}/{batch}/{batch}客户类型样本统计表.xlsx`

说明：

- 样本汇总中涉及默认年份时，统一使用 CLI 传入的 `--year`

### 14.9 PPT 生成

输入：

- `data/satisfaction_detail/{year}/{batch}`

输出：

- `data/ppt/{year}/{batch}/{batch}满意度报告.pptx`

## 15. 日志与失败恢复

### 15.1 日志目录

日志目录固定为：

```text
logs/pipeline/{year}/{batch}/
```

### 15.2 日志文件

至少生成：

- `pipeline.log`
- `precheck.log`
- `unmapped_customer_records.log`

### 15.3 失败恢复策略

本阶段采用简单可靠策略：

1. 预查错失败：暂停等待人工修正
2. 正式执行失败：立即停止
3. 保留已生成中间结果和日志
4. 下次重新运行同一命令时，默认覆盖重算

本阶段不实现复杂断点续跑。

## 16. 兼容与迁移策略

### 16.1 保留旧能力

保留：

- 旧脚本 CLI 入口
- 旧批次配置文件
- GUI 现有流程

### 16.2 新入口定位

新增主流程 CLI 作为推荐新入口。  
待运行稳定后，再评估是否逐步废弃：

- `job*.toml`
- `report_jobs*.toml`

### 16.3 迁移顺序

建议迁移顺序：

1. 建立 `data/...` 目录
2. 把现有批次原始数据迁入 `data/raw/{year}/{batch}`
3. 使用新主流程命令跑通
4. 旧 TOML 跑法保留一段时间作为回退方案

## 17. 测试策略

本项目要求采用测试驱动开发，Phase 1 设计如下测试范围。

### 17.1 单元测试

1. 路径解析测试
   - `year + batch` 能正确推导各目录和文件路径
2. 默认配置加载测试
   - 全局默认配置能正确读取
   - 缺失字段能正确报错或使用默认值
3. 预查错分类测试
   - 阻断问题与警告问题分类正确
4. 确认输入测试
   - `y / yes / 继续` 可识别
   - 其他输入不会误触发继续

### 17.2 编排测试

通过 mock 现有业务引擎函数验证主流程顺序：

1. 预查错失败 -> 等待确认 -> 重新检查 -> 成功后继续
2. 预查错失败 -> 重新检查仍失败 -> 继续等待
3. 正式执行阶段某一步失败 -> 整体停止

### 17.3 轻量集成测试

验证：

1. 主流程可以驱动 `survey_stats.py`、`summary_table.py`、`sample_table.py`、`generate_ppt.py` 的函数式入口
2. 输出路径与文件名符合约定

### 17.4 测试执行方式

继续统一使用：

```bash
uv run pytest
```

## 18. 风险与控制措施

### 18.1 风险：现有脚本内部入口风格不一致

控制措施：

- 主流程只抽取真正需要的函数式入口
- 不一次性重构所有脚本结构

### 18.2 风险：用户数据目录未严格遵守标准文件名

控制措施：

- 预查错阶段明确检查并报阻断
- 不做模糊猜测

### 18.3 风险：查错通过后仍可能在后续脚本阶段失败

控制措施：

- 完整保留日志
- 保留中间结果
- 支持重复运行覆盖重算

## 19. 实施建议

Phase 1 推荐实施顺序：

1. 抽取路径与默认配置层
2. 抽取预查错封装层
3. 新增 `main_pipeline.py`
4. 为现有引擎补齐最小函数式入口
5. 补主流程测试
6. 用一个真实批次做端到端验证

## 20. 最终结论

本设计采用：

- 固定目录约定
- 固定标准来源文件名
- 一份全局默认配置
- 一个 CLI 主流程入口
- 查错暂停、确认后复查再继续
- 复用现有业务引擎脚本

这是在当前代码库基础上，风险最低、收益最大的第一阶段优化方案。
